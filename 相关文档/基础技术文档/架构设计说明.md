# 架构设计说明

## 1. 整体架构

LabelHub 采用前后端分离的单体应用架构。前端 React SPA 通过 Nginx 反向代理访问 FastAPI 后端，中间件（MySQL/Redis/RabbitMQ）Docker 容器化运行。

**为什么不用微服务？** 项目规模适中，单机部署满足需求，微服务会增加不必要的网络开销和运维复杂度。

## 2. Schema Designer — 动态表单架构

### 问题

标注任务需要灵活的表单模板。如果用传统方式手写表单组件，每个新任务都需要前端改代码重新部署。

### 方案

可视化表单搭建器，拖拽生成 JSON Schema，运行态由 Formily 引擎动态渲染。

**三层数据模型**：Tab（页签）→ Group（容器）→ Field（物料）

**Designer / Renderer 解耦**：设计态产物是标准 JSON Schema，可序列化存储、版本管理、导入导出。运行态 Labeler 标注工作台使用同一套 Formily 渲染引擎，一处搭建处处生效。

**技术细节**：
- 状态管理：useReducer 集中管理 fields/groups/tabs + selectedFieldId/selectedGroupId
- 拖拽交互：@dnd-kit 实现物料面板→画布拖放 + Tab 排序 + 容器内字段排序
- 字段联动：条件显示/隐藏，基于目标字段值动态控制
- 10 种物料：单行/多行/单选/多选/下拉/日期/富文本/JSON/LLM交互/ShowItem

## 3. AI Agent 池架构

### 问题

AI 审核需要高并发处理，每次创建新 OpenAI Client 会导致冷启动延迟。配置变更需要重启服务。

### 方案

线程池 + 长生命周期实例。

**执行流程**：Labeler 提交 → Redis FIFO 入队 → RabbitMQ 发布 → ai_worker 消费 → AgentPool.submit() → Agent.run()

**设计要点**：
- 长生命周期：Agent 启动时加载 System Prompt + 评分维度，后续调用零冷启动
- 线程池并发：AGENT_POOL_SIZE 控制最大并发数
- 配置热重载：pool.reload() 增量同步 DB 变更，不打断工作中的 Agent
- MQ 降级：RabbitMQ 不可用时自动跳过，标注提交不受阻
- Agent 即用户：每个 AI Agent 是独立 `ai_agent` 角色 User，审核记录归属清晰

**为什么不用 Celery？** 项目单机部署，Agent < 10 个，Celery 过于复杂且运维负担重。

## 4. 抢单架构

### 问题

多标注员同时认领同一任务时，如何保证不超发？

### 方案

Redis SETNX 分布式锁 + DB 二次校验。

**三种策略**：
| 策略 | 锁粒度 | 并发方式 |
|------|--------|---------|
| first_come | task 级 Redis 锁 | 串行，每次分配 1 条 |
| quota_grab | item 级 DB 行锁 (SKIP LOCKED) | 并行，各抢各的 item |
| assigned | 无锁 | Owner 手动分配 |

**两阶段防竞态**：Redis SETNX 快速拦截（30s 锁超时）→ DB WHERE claimed_items < grab_limit 精确兜底。

## 5. 状态机设计

三级状态机覆盖完整业务流程：

| 级别 | 状态数 | 转移通道 |
|------|--------|---------|
| Task | 4 态 | draft→published⇄paused→ended |
| Item | 3 态 | pending→labeled/skipped, skipped→labeled |
| Result | 6 态 | submitted→ai_reviewing→review→final_review→warehouse + rejected |

**设计原则**：
- VARCHAR 存储状态值（禁用 ENUM），枚举上浮到应用层
- validate_transition() 校验所有转移合法性，非法转移自动拒绝
- 每次迁移写入 audit_logs，完整审计时间线可追溯
