# 前后端技术选型说明

## 前端选型

### 为什么选 React 19 + TypeScript？

- **React 19**：函数组件 + Hooks 生态成熟，社区资源丰富。19 版本支持 Server Components 但项目仅用客户端渲染。
- **TypeScript strict**：全项目零 `any`，所有类型集中在 `types/models/` 管理，编译期拦截类型错误。
- **Vite 8**：相比 Webpack，冷启动 < 1s，HMR 即时生效，构建产物按需拆分。

### 为什么选 Ant Design 6 + Formily？

- **Ant Design 6**：企业级 UI 组件库，Table/Form/Modal/Drawer 完善，移动端响应式内置。
- **Formily**：JSON Schema 驱动表单渲染，解决了 Schema Designer 搭建产物到 Labeler 运行态的同源渲染问题。不用 Formily 的话需要自己写一套表单引擎。

### 为什么选 Zustand？

- 比 Redux 轻量：无 boilerplate，直接 `create( (set,get) => ({...}) )`
- 比 Context 高效：按需订阅，避免不相关组件重渲染
- 比 MobX 简单：不引入装饰器和 observable 概念

### 为什么选 @dnd-kit？

- 现代拖拽库，支持键盘操作和屏幕阅读器
- `SortableContext` 支持水平和垂直排序
- 比 react-beautiful-dnd 更活跃维护

## 后端选型

### 为什么选 FastAPI？

- **异步原生**：async/await 写法，与 uvicorn 配合性能优秀
- **自动文档**：Swagger UI + ReDoc 自动生成，无需手写 API 文档
- **Pydantic 集成**：请求/响应自动校验，类型安全

### 为什么选 SQLAlchemy 2.0？

- ORM 声明式模型定义，迁移方便
- 2.0 支持 `mapped_column` 新语法，类型提示更准确
- `create_all` 自动建表，省去单独 migration

### 为什么选 RabbitMQ？

- 持久化消息：队列和消息都标记 durable，重启不丢失
- 手动 ACK：consumer 处理完才确认，异常时消息重新入队
- 比 Celery 简单：直接 pika 连接，不需要额外的 broker 配置

### 为什么选双 Token JWT？

- **Access Token (30min)**：无状态，不查 Redis，高性能验证
- **Refresh Token (7d)**：jti 白名单存 Redis，可吊销
- **一次性 Refresh**：刷新时旧 jti 立即删除，新 jti 入白名单，防止泄露后被长期利用

## 为什么不用 Docker 全容器化部署？

| 考量 | 决策 | 理由 |
|------|------|------|
| 后端部署 | 宿主机裸跑 + systemd | 频繁更新代码时 `git pull + systemctl restart` 比 `docker build + up` 快得多 |
| 前端部署 | Nginx 静态托管 | Vite build 产物是纯静态文件，不需要 Node 运行时 |
| 中间件 | Docker 容器 | MySQL/Redis/RabbitMQ 配置稳定，Docker 管理省心 |
| SSL | Nginx Docker | 统一 SSL 终端，Let's Encrypt 免费证书 |
