# 状态机并发一致性 — 场景分析与方案

## 场景 1：两个标注员同时领取最后一题（配额超发）

**问题**：任务配额 1，A 和 B 同时点"领取"。都读到 `completed_items=9, total_items=10`，都认为还能领。

**方案：Redis SETNX 分布式锁**

```python
# services/tasks/service.py

def claim_task(db, task_id, user_id):
    lock_key = f"lock:claim:{task_id}:{user_id}"
    lock = RedisLock(lock_key, ttl=5)
    
    if not lock.acquire():
        raise ConflictException("正在处理中，请稍后重试")
    
    try:
        task = db.query(LabelTask).filter_by(id=task_id).first()
        
        # 双重检查（获取锁后再查一次状态）
        if task.status != TaskStatus.PUBLISHED:
            raise BadRequestException("任务未在发布状态")
        
        # 检查配额
        claimed = db.query(func.count(TaskItem.id)).filter(
            TaskItem.task_id == task_id,
            TaskItem.status.in_(["labeled", "pending"]),
        ).scalar()
        
        if claimed >= task.total_items:
            raise BadRequestException("配额已满")
        
        # 分配题目
        item = db.query(TaskItem).filter(
            TaskItem.task_id == task_id,
            TaskItem.status == "pending",
            TaskItem.labeler_id.is_(None),
        ).first()
        
        if not item:
            raise BadRequestException("没有待分配题目")
        
        item.labeler_id = user_id
        db.commit()
        
    finally:
        lock.release()
```

**为什么用 Redis 锁而不是数据库锁？**
- 领单是高频操作（标注员频繁刷新），Redis 锁比数据库悲观锁（SELECT FOR UPDATE）轻量
- TTL=5s 防止死锁（进程崩溃后锁自动释放）

## 场景 2：AI Agent 审核中，Reviewer 同时打回

**问题**：Result 状态为 `ai_reviewing`，Reviewer 看到了然后点"打回"。

**方案：状态机本身的校验就够**

```python
# 审核时先校验当前状态
def review_result(db, result_id, action, comment):
    result = db.query(LabelResult).filter_by(id=result_id).first()
    
    # 状态机校验：ai_reviewing 不能直接 rejected
    # Reviewer 的通过/打回只在 review/final_review 阶段可用
    if action == "reject":
        if not validate_transition("LabelResult.status", result.status, "rejected"):
            raise ConflictException(f"当前状态 {result.status} 不允许打回操作")
    
    # 执行状态转移
    new_status = get_target_status(result.status, action)
    transit(result, "status", "LabelResult.status", new_status, db, actor_info)
    db.commit()
```

**问题**：Reviewer 在 `ai_reviewing` 阶段根本看不到可操作的按钮（前端根据状态动态显示按钮），所以这个冲突在大部分情况下不会发生。即使 Reviewer 通过 API 直接调用，状态机校验也会拦截。

## 场景 3：两个 Reviewer 同时对同一题做终审

**问题**：A 和 B 同时点"通过 final_review"，都想转到 warehouse。

**方案：状态机 + 事务原子性**

```python
def final_review_approve(db, result_id, reviewer_id):
    result = db.query(LabelResult).filter_by(id=result_id).first()
    
    # 关键：在同一个事务中校验状态
    if result.status != ResultStatus.FINAL_REVIEW:
        raise ConflictException("当前状态不允许此操作，请刷新")
    
    if not validate_transition("LabelResult.status", result.status, "warehouse"):
        raise ConflictException("非法状态转移")
    
    # 执行转移
    transit(result, "status", "LabelResult.status", "warehouse", db, {
        "id": reviewer_id, "name": "...", "role": "reviewer"
    })
    db.commit()
```

由于数据库事务的 ACID 特性，A 和 B 的操作是串行化的。先到的事务成功，后到的事务读到 `status=warehouse`，状态机校验拒绝。

**不需要额外锁**：数据库事务本身就提供了隔离性保证。

## 场景 4：Owner 暂停任务时，有标注员正在提交

**问题**：任务暂停后，已有的标注提交该怎么处理？

**方案：分情况处理，不拒绝合法操作**

```python
# Owner 暂停任务
def pause_task(db, task_id, owner_id):
    task = db.query(LabelTask).filter_by(id=task_id).first()
    transit(task, "status", "LabelTask.status", "paused", db, actor_info)
    db.commit()

# Labeler 提交标注（暂停后仍允许提交已领取的题目）
def submit_result(db, task_id, item_id, user_id, data):
    task = db.query(LabelTask).filter_by(id=task_id).first()
    
    # paused 状态允许提交已领取的题目，但拒绝新领取
    if task.status == TaskStatus.ENDED:
        raise BadRequestException("任务已结束，无法提交")
    
    # paused 状态下正常处理提交
    result = LabelResult(task_id=task_id, item_id=item_id, labeler_id=user_id, data=data, status="submitted")
    db.add(result)
    db.commit()
```

**设计决策**：暂停任务只影响"新领取"，不影响"已有提交"。这样标注员正在做的题目不会白费。

## 场景 5：Labeler 修改被达回的提交时，AI Agent 重跑审核

**问题**：Labeler 点了"重新提交"（status 从 rejected → submitted），同时 Reviewer 点了"重跑 AI 审核"。

**方案：重提交时重置状态，旧 AI Review 标记为非最新**

```python
# Labeler 重新提交
def resubmit(db, result_id, new_data):
    result = db.query(LabelResult).filter_by(id=result_id).first()
    
    if result.status != ResultStatus.REJECTED:
        raise ConflictException("只有被打回的题目可以重新提交")
    
    result.data = new_data
    result.round += 1  # 轮次 +1
    result.status = ResultStatus.SUBMITTED  # 回到提交状态
    db.commit()
    
    # 重新触发 AI 审核
    try:
        mq_publish("labelhub.ai.review", {
            "task_id": result.task_id,
            "item_id": result.item_id,
            "result_id": result.id,
        })
    except:
        pass

# AI 审核写入时，新结果追加行（不覆盖旧记录）
# AiReview 表用 result_id + round 联合标识
```

| 并发场景 | 方案 | 理由 |
|---------|------|------|
| 领单超发 | Redis SETNX 锁 | 高频操作，Redis 锁轻量 |
| AI 审核 vs Reviewer | 状态机校验 | 人工操作低频，状态机足够 |
| 双 Reviewer 终审 | DB 事务串行化 | 不需要额外锁 |
| 暂停 vs 提交 | 容忍继续提交 | 不浪费已有工作 |
| 修改 vs 重审 | 状态重置 + 追加记录 | AiReview 独立表天然支持 |
