# 后端技术栈评估

## 我的提示词

```
你是一个后端架构师，我需要为 LabelHub 数据标注平台选择后端技术栈。

项目关键需求：
- RESTful API 接口（10+ 组端点）
- AI Agent 异步审核流水线（LLM 调用 + 结果写入）
- 多级状态机（任务/题目/提交/审核四级状态流转）
- 数据导出（JSON/JSONL/CSV/Excel，异步）
- 文件上传（图片/附件上传至 OSS）
- 完整的审计日志

请帮我评估：
1. 框架：FastAPI vs Django REST vs Flask — 对比异步支持、数据校验、开发效率
2. ORM：SQLAlchemy vs Django ORM vs Tortoise ORM — 哪个更适合复杂状态机
3. 任务队列：Celery vs RabbitMQ 原生 vs Redis Queue — 对于 Agent 审核场景哪个合适
4. 缓存/锁：Redis 在项目中的应用场景（分布式锁/队列/缓存）
5. 认证方案：JWT 双 Token 方案的实现细节
6. 为什么不用 LangChain 而要直接用 OpenAI SDK

输出：推荐技术栈 + 每项给出 2-3 条选择理由
```

---

## AI 的回答

### 1. 框架：FastAPI

**结论：Python FastAPI**

| 对比 | FastAPI | Django REST | Flask |
|------|---------|-------------|-------|
| 异步支持 | ✅ 原生 async/await | ⚠️ 3.x 开始支持 | ❌ 需扩展 |
| 数据校验 | ✅ Pydantic 深度集成 | Serializer 手动写 | ❌ 需手动 |
| 自动文档 | ✅ /docs Swagger 自动生成 | 需 drf-spectacular | ❌ 需手写 |
| LLM 调用适配 | ✅ async OpenAI SDK | 需配置 | With gevent |

理由：FastAPI 原生异步 + Pydantic 校验，对于"异步调用 LLM + 结构化输出校验"的 AI Agent 场景适配最佳。自动生成的 Swagger 文档也是加分项。

### 2. ORM：SQLAlchemy 2.0

**结论：SQLAlchemy 2.0**

理由：
- 状态机需要精细的事务控制（开启→校验状态→业务逻辑→提交→写审计日志），SQLAlchemy 的 session 管理灵活且可控
- 与 FastAPI 搭配最成熟（Depends(get_db) 模式）
- Django ORM 必须绑定 Django 项目，不能单独使用
- Tortoise ORM 异步方案好但生态小，遇到复杂查询时文档少

### 3. 任务队列：RabbitMQ 原生 + 线程池

**结论：不用 Celery，用 RabbitMQ + ThreadPoolExecutor**

理由：
- Celery 太重：需要独立 worker 进程 + broker + result backend，项目 Agent 数 < 10 不需要这套分布式
- 线程池方案更轻量：Agent 作为有状态实例常驻内存（持有自己的 OpenAI Client + prompt 配置），零冷启动
- RabbitMQ 做消息解耦：标注提交 → MQ → worker 消费 → pool.submit(agent.run)，链路清晰
- 配置热重载：pool.reload() 增量同步 DB 变更，Celery task 是无状态的做不到

### 4. 缓存/锁：Redis 三大场景

| 场景 | Redis 用法 | 说明 |
|------|-----------|------|
| 并发领单防超发 | SETNX 分布式锁 | claim 接口获取锁，锁 key = `task:{id}:claim:{user_id}` TTL=5s |
| JWT Refresh Token 白名单 | SET + GET + DEL | refresh_token 的 jti 写入 Redis，refresh 时校验存在并删除（一次性消费） |
| Schema JSON 缓存 | GET + SET with TTL | 频繁读取的 Schema JSON 缓存，TTL=5min，更新时失效 |
| 简化队列 | BLPOP/BRPOP | MQ 不可用时降级使用 Redis Queue |

### 5. 认证：JWT 双 Token 方案

**Access Token（30 分钟）**：
- 无状态验证（不查 Redis，用 JWT secret 解码）
- payload: `{sub: user_id, role: "owner", type: "access", exp: ...}`

**Refresh Token（7 天）**：
- 有状态验证（jti 存在 Redis 白名单中）
- payload: `{sub: user_id, jti: uuid, type: "refresh", exp: ...}`
- 每次 refresh 时：校验 jti 在 Redis 中存在 → 删除旧 jti → 签发新 jti 写入 Redis → 返回新 token pair
- 退出登录时删除 Redis 中 jti

前端 401 自动处理：
- Axios 响应拦截器：401 → 用 refresh_token 换新 access_token → 静默重试原请求
- 刷新期间其他并发请求排队等待同一个 refresh Promise（防并发刷新）

### 6. 为什么不用 LangChain

**直接用 OpenAI SDK（openai 包）**

理由：
- LangChain 抽象层太厚：Agent 场景只需要 "构建 prompt → 调 API → 解析 JSON"，不需要 Chain/Agent/Tool 这些抽象
- LangChain 版本更新频繁 API 不稳定：v0.1 → v0.2 → v0.3 每个版本都有 breaking change
- OpenAI SDK 足够：`client.chat.completions.create(model=..., messages=[...], response_format={type:"json_object"})` 三行代码
- Pydantic 校验自己做：`ReviewResult.model_validate(response)` 比 LangChain 的 `with_structured_output` 更灵活可调试

### 最终技术栈

| 层 | 选择 |
|----|------|
| 框架 | FastAPI |
| ORM | SQLAlchemy 2.0 + PyMySQL |
| 数据校验 | Pydantic v2 + pydantic-settings |
| 任务队列 | RabbitMQ (pika) + ThreadPoolExecutor |
| 缓存/锁 | Redis (redis-py) |
| 认证 | PyJWT + bcrypt (passlib) |
| LLM SDK | openai |
| 测试 | pytest + httpx |
