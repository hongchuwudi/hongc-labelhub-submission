# 中间件选型

## 我的提示词

```
LabelHub 项目还需要基础设施中间件，请帮我做选型评估：

1. 数据库：MySQL vs PostgreSQL — 标注平台的数据特点更适合哪个
2. 消息队列：RabbitMQ vs Kafka vs Redis Stream — 对于 AI 审核 + 导出两个异步场景
3. 缓存：Redis 的使用场景分析（JWT 白名单、分布式锁、Schema 缓存、并发控制）
4. 文件存储：阿里云 OSS vs MinIO vs 本地存储
5. 部署方案：Docker Compose vs K8s — 比赛/演示阶段选哪个

输出：推荐方案 + 每项（一句话结论 + 一句话理由）
```

---

## AI 的回答

### 1. 数据库：MySQL 8.0

**结论：MySQL 8.0**

- 一句话结论：MySQL，因为标注数据以结构化 JSON 字段 + 关系型查询为主，不需要 PG 的高级特性
- 理由：LabelHub 的数据模型是典型的关系型（任务-题目-提交-审核多表关联），MySQL 的事务和 JOIN 性能足够。JSON 字段 MySQL 8.0 已支持 JSON 类型 + 索引。中标场合大部分云平台 MySQL 更方便。

### 2. 消息队列：RabbitMQ

**结论：RabbitMQ 3.x**

- 一句话结论：RabbitMQ，因为需要 ACK 确认 + 持久化保证消息不丢，Redis Stream 消费组管理不如 RabbitMQ 成熟
- 理由：AI 审核场景关键是"每条提交必须被审核"，RabbitMQ 的 ack 机制保证消息不丢。导出场景不需要 Kafka 的高吞吐。一个 RabbitMQ 实例同时服务两个队列（ai.review + export），运维简单。

### 3. 缓存/锁：Redis

**结论：Redis 7.x**

使用场景：

| 场景 | Key 设计 | TTL | 说明 |
|------|---------|-----|------|
| Refresh Token 白名单 | `jti:{uuid}` | 7天 | refresh 时验证 → 删除 → 重新签发 |
| 领单分布式锁 | `lock:claim:{task_id}:{user_id}` | 5秒 | SETNX 防并发超发 |
| Schema 缓存 | `schema:{id}` | 5分钟 | 频繁读取的 Schema JSON |
| 降级消息队列 | `queue:ai.review` | — | MQ 不可用时 BLPOP 降级 |
| 限流 | `rate:{user_id}:{endpoint}` | 每分钟 | 防刷 |

### 4. 文件存储：阿里云 OSS

**结论：阿里云 OSS**

- 一句话结论：OSS，因为比赛/演示阶段用云存储免运维，导出文件需要公网可下载
- 理由：导出文件需要给用户下载链接，本地存储需要额外搭 Nginx 文件服务。OSS 自带 CDN + 签名 URL，3 行代码生成临时下载链接。MinIO 需要自己维护，对比赛场景太重。

### 5. 部署方案：Docker Compose

**结论：Docker Compose**

- 一句话结论：Docker Compose，因为比赛阶段单机部署，一个 `docker-compose up -d` 启动全部服务
- 理由：K8s 对 5 个服务（应用 + MySQL + Redis + RabbitMQ + Nginx）是过度设计。docker-compose.yml 统一定义所有服务 + 网络 + 卷，评测老师一条命令复现。

### 最终中间件清单

| 中间件 | 选型 | 用途 |
|--------|------|------|
| 数据库 | MySQL 8.0 | 业务数据（10 张表） |
| 消息队列 | RabbitMQ 3.x | AI 审核队列 + 导出队列 |
| 缓存/锁 | Redis 7.x | JWT 白名单/分布式锁/Schema 缓存 |
| 文件存储 | 阿里云 OSS | 文件上传 + 导出文件 |
| 部署 | Docker Compose | 一键启动全部服务 |
