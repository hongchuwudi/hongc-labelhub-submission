# 前端技术栈评估

## 我的提示词

```
你是一个前端架构师，我需要为 LabelHub 数据标注平台选择前端技术栈。

项目关键需求：
- 可视化拖拽搭建表单模板（核心难点）
- 动态表单渲染（Schema 驱动）
- 复杂状态管理（任务/审核多状态流转）
- 三种角色的独立工作台
- 移动端适配（可选）
- TypeScript 全栈，禁止 any

请帮我评估并推荐技术选型：
1. 框架：React vs Vue vs Angular — 为什么选 React
2. UI 组件库：Ant Design vs Semi Design vs Arco Design — 对比优缺点
3. 表单方案：Formily vs react-hook-form vs formik — 哪个更适合 Schema 驱动的动态表单
4. 拖拽库：dnd-kit vs react-dnd vs react-beautiful-dnd — 哪个更适合表单搭建器
5. 状态管理：Zustand vs Jotai vs Redux Toolkit — 中小团队选哪个
6. 路由方案：react-router v6/v7 的 Data API 是否采用

输出：推荐技术栈 + 每项 3 句话对比理由
```

---

## AI 的回答

### 1. 框架选型：React 18

**结论：React 18 + TypeScript**

| 对比 | React | Vue 3 | Angular |
|------|-------|-------|---------|
| 生态 | Formily 对 React 支持最好 | 表单方案少 | 过重 |
| 拖拽 | dnd-kit 专为 React 设计 | 需 Vue 封装 | CDK 学习成本高 |
| 类型 | TSX 类型推断成熟 | 模板类型支持弱 | 天然 TypeScript 但笨重 |

理由：Formily + dnd-kit 两个核心依赖都深度绑定 React，生态匹配度最高。Vue 的模板语法不适合高度动态的表单场景。

### 2. UI 组件库：Ant Design 6

**结论：Ant Design 6**

| 对比 | Ant Design 6 | Semi Design | Arco Design |
|------|-------------|-------------|-------------|
| 组件丰富度 | 极全（Form/Table/Modal/Timeline/Upload 全场景） | 中等 | 中等 |
| Table 能力 | 功能最强（排序/筛选/展开行） | 基本够用 | 基本够用 |
| 文档中文 | 最完善 | 好 | 好 |
| Formily 适配 | @formily/antd 官方维护 | 无官方适配 | 无官方适配 |

理由：Ant Design 组件最全（特别是 Table 和 Form），且 Formily 官方提供 @formily/antd 桥接，适配成本最低。

### 3. 表单方案：Formily

**结论：Formily 2.x**

| 对比 | Formily | react-hook-form | formik |
|------|---------|-----------------|--------|
| Schema 驱动 | ✅ 核心能力，JSON Schema → 渲染 | ❌ 需手写 | ❌ 需手写 |
| Designer/Renderer 解耦 | ✅ 天然支持 | ❌ | ❌ |
| 联动（reactions） | ✅ x-reactions 声明式 | ❌ 需 watch + useEffect | ❌ 需手动 |
| 复杂表单性能 | ✅ 字段级响应 | ✅ 非受控（快） | ❌ 全量重渲染 |

理由：Formily 是唯一支持"JSON Schema 定义 → 引擎渲染"的方案，与 LabelHub 的 Designer/Renderer 解耦需求完美匹配。x-reactions 声明式联动比 useEffect 维护性强太多。

### 4. 拖拽库：@dnd-kit

**结论：@dnd-kit/core + @dnd-kit/sortable**

| 对比 | @dnd-kit | react-dnd | react-beautiful-dnd |
|------|----------|-----------|---------------------|
| 维护状态 | ✅ 活跃 | ⚠️ 停滞 | ❌ 已归档 |
| 自定义拖拽覆盖层 | ✅ DragOverlay 完美支持 | 需 hack | 不支持 |
| 可访问性 | ✅ 内置键盘操作 | ❌ | ✅ |
| 多容器嵌套 | ✅ 碰撞检测算法灵活 | 复杂 | 有限 |

理由：@dnd-kit 是唯一还在活跃维护的拖拽库。DragOverlay 对于"从物料面板拖一个字段卡片到画布上"的场景体验极好，碰撞检测算法支持容器嵌套判断（字段是否拖入分组容器内）。

### 5. 状态管理：Zustand

**结论：Zustand 5**

| 对比 | Zustand | Jotai | Redux Toolkit |
|------|---------|-------|---------------|
| 上手成本 | 极低（3 分钟学会） | 低 | 高（slice/thunk/selector） |
| 体积 | 1KB | 2KB | 12KB |
| 跨组件读取 | ✅ useStore(state => state.user) | ✅ atom 共享 | ✅ useSelector |
| DevTools | ✅ | ✅ | ✅ |

理由：LabelHub 全局只需要存 user/token/authReady 少量状态，Redux 的样板代码是过度设计。Zustand 的 API 简单到只需要一个 `create((set) => ({...}))`。

### 6. 路由：react-router-dom v7（普通模式）

**结论：使用传统 <Routes> 模式，不采用 Data API**

理由：Data API（loader/action）适合 SSR/数据预取场景，LabelHub 是纯 SPA + Axios 请求数据，用 Data API 反而把数据获取逻辑拆散到路由定义和组件两处，增加复杂度。

### 最终技术栈

| 层 | 选择 |
|----|------|
| 框架 | React 19 + TypeScript 5.7 |
| UI | Ant Design 6 |
| 表单 | Formily 2.x (core + react + antd) |
| 拖拽 | @dnd-kit/core + @dnd-kit/sortable + @dnd-kit/utilities |
| 状态 | Zustand 5 |
| 路由 | react-router-dom v7 |
| 图表 | ECharts + echarts-for-react |
| 富文本 | react-quill-new |
| JSON 编辑器 | @monaco-editor/react |
