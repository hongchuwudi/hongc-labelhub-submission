# 动态表单架构 — 难点解决方案

## 1. 三层数据模型：Tab → Group → Field

```typescript
// types.ts — 类型定义

/** 字段配置 — 叶子节点 */
interface FieldConfig {
  id: string;            // uuid
  type: FieldType;       // 'text' | 'textarea' | 'radio' | 'checkbox' | ...
  key: string;           // 表单字段 key
  title: string;         // 显示标题
  required?: boolean;
  defaultValue?: unknown;
  placeholder?: string;
  options?: { label: string; value: string }[];  // 单选/多选用
  validations?: ValidationRule[];                 // 校验规则
  llmConfig?: LLMConfig;                         // LLM 交互配置
  tabId?: string;                                // 所属 Tab ID
  groupId?: string;                              // 所属 Group ID
}

/** 分组容器 — 中间层 */
interface GroupConfig {
  id: string;
  title: string;
  fieldIds: string[];   // 包含的字段 ID 列表
  collapsible?: boolean;
  collapsed?: boolean;
}

/** Tab 页 — 顶层 */
interface TabConfig {
  id: string;
  title: string;
  fieldIds: string[];   // 直接包含的字段
  groupIds: string[];   // 包含的分组
}

/** 全局 Schema 设计状态 */
interface SchemaDesign {
  fields: Record<string, FieldConfig>;  // 所有字段（平铺存储）
  groups: Record<string, GroupConfig>;  // 所有分组
  tabs: TabConfig[];                    // 所有 Tab（有顺序）
}
```

**设计要点：**
- FieldConfig 中 `tabId` 和 `groupId` 互斥（字段要么在 Tab 下要么在 Group 下）
- 字段在 Group 内时，`tabId` 为空，继承 Group 所在 Tab
- 平铺存储 + ID 引用，避免深层嵌套带来的更新复杂性

## 2. Designer / Renderer 解耦

**Schema 序列化为 Formily JSON Schema：**

```typescript
// utils/schemaTransformer.ts — SchemaDesign → Formily Schema

function toFormilySchema(design: SchemaDesign): FormilySchema {
  const properties: Record<string, any> = {};
  
  for (const tab of design.tabs) {
    const tabProperties: Record<string, any> = {};
    
    // 渲染 Tab 下的字段
    for (const fieldId of tab.fieldIds) {
      const field = design.fields[fieldId];
      tabProperties[field.key] = fieldToFormilySchema(field);
    }
    
    // 渲染 Tab 下的分组
    for (const groupId of tab.groupIds) {
      const group = design.groups[groupId];
      const groupProperties: Record<string, any> = {};
      for (const fieldId of group.fieldIds) {
        const field = design.fields[fieldId];
        groupProperties[field.key] = fieldToFormilySchema(field);
      }
      tabProperties[group.title] = {
        type: 'void',
        'x-component': 'FormLayout',
        properties: groupProperties,
      };
    }
    
    properties[tab.title] = {
      type: 'void',
      'x-component': 'FormTab.TabPane',
      'x-component-props': { tab: tab.title },
      properties: tabProperties,
    };
  }
  
  return {
    type: 'object',
    properties: {
      layout: {
        type: 'void',
        'x-component': 'FormTab',
        properties,
      },
    },
  };
}
```

**关键点：** Designer 存自己的数据模型（Tab/Group/Field），Renderer 用 Formily Schema 渲染。两者通过 `schemaTransformer` 桥接，中间那层 JSON Schema 就是可序列化的产物。

## 3. dnd-kit 拖拽交互

**场景分类：**

| 场景 | dnd-kit 机制 | 实现要点 |
|------|-------------|---------|
| 物料面板→画布 | `DndContext.onDragEnd` | 判断 `over` 是否在 Canvas 区域，创建新 FieldCard |
| 容器内排序 | `SortableContext` | 字段卡片在分组容器内排序 |
| Tab 间拖拽 | 切换 Tab 时切换 `SortableContext` | 拖入不同 Tab → 更新字段的 `tabId` |
| 拖入分组容器 | `collisionDetection` | rectIntersection 检测卡片是否在 Group 区域内 |

```tsx
// index.tsx — 核心 dragEnd 处理
function handleDragEnd(event: DragEndEvent) {
  const { active, over } = event;
  
  // 1. 从物料面板拖入新字段
  if (active.data.current?.type === 'palette-item') {
    const fieldType = active.data.current.fieldType;
    const newField = createField(fieldType);
    
    if (isOverGroup(over)) {

      dispatch({ type: 'ADD_FIELD', field: newField });

      dispatch({ type: 'SET_FIELD_TAB', fieldId: newField.id, tabId: over.id });

}
```

## 4. 字段联动 — x-reactions

```typescript
// 在 SchemaDesign 中存储联动配置
interface FieldConfig {
  // ...其他字段
  reactions?: ReactionConfig[];
}

interface ReactionConfig {
  type: 'visibility' | 'value' | 'validate';
  condition: {
    fieldKey: string;      // 依赖的字段
    operator: 'eq' | 'neq' | 'gt' | 'lt' | 'contains';
    value: unknown;
  };
  target?: unknown;        // 联动目标值
}

// 转换到 Formily Schema 时生成 x-reactions
function fieldToFormilySchema(field: FieldConfig) {
  const schema: any = { /* ... */ };
  
  if (field.reactions?.length) {
    schema['x-reactions'] = field.reactions.map(r => {
      if (r.type === 'visibility') {
        return {
          dependencies: [r.condition.fieldKey],
          fulfill: {
            state: {
              visible: `{{$deps[0] ${getOperator(r.condition.operator)} ${JSON.stringify(r.condition.value)}}}`
            }
          }
        };
      }
    });
  }
  
  return schema;
}
```

## 5. 校验规则序列化

```typescript
interface ValidationRule {
  type: 'required' | 'minLength' | 'maxLength' | 'pattern' | 'custom';
  message: string;     // 错误提示
  value?: unknown;     // 校验参数
}

// 转换到 Formily x-validator
function toFormilyValidator(validations: ValidationRule[]) {
  const rules: any[] = [];
  
  for (const v of validations) {
    switch (v.type) {
      case 'required':
        rules.push({ required: true, message: v.message });
        break;
      case 'minLength':
        rules.push({ min: v.value, message: v.message });
        break;
      case 'pattern':
        rules.push({ pattern: new RegExp(v.value as string), message: v.message });
        break;
    }
  }
  
  return rules;
}
```

## 6. 版本管理

**设计：任务发布时冻结 Schema 快照**

```
LabelSchema 表
├── schema: JSON        # 当前最新版本的 Schema（可编辑）
├── version: INT         # 版本号
└── version_history: [   # 历史版本
      { version: 1, schema: {...}, created_at: "..." },
      { version: 2, schema: {...}, created_at: "..." },
    ]

LabelTask 表
└── schema_snapshot: JSON  # 发布时复制 schema 的当前版本
```

- 任务发布后，Owner 可以继续修改 Schema（version+1），不影响已发布的任务
- 标注员作答时读取 `task.schema_snapshot`，保证表单结构不变
- 导入旧版本可回滚

## 7. 大表单性能优化

| 优化策略 | 说明 |
|---------|------|
| Tab 懒渲染 | 只渲染当前激活 Tab 的字段，切换时按需渲染 |
| Formily 字段级响应 | Formily 天然支持字段级响应式更新，不会全表单重渲染 |
| 分组折叠默认 | 父容器默认折叠，减少初次渲染的 DOM 数量 |
| Monaco Editor 懒加载 | `React.lazy(() => import('@monaco-editor/react'))` |
| 草稿防抖 | 字段变更后 3s debounce 再保存 |
| 虚拟滚动 | ItemQueue 题目列表超 100 条时用虚拟滚动 |
